# 自动工作流程策略

**创建时间**: 2026-03-29 06:27 PDT
**状态**: ✅ 激活

---

## 🎯 核心策略

**自动完成模式** - 所有新认领的任务自动按顺序完成，无需询问用户

---

## 📋 工作流程

### 1. 扫描阶段
- 每30分钟自动扫描（Bash + Python扫描器）
- 自动评估任务价值（评分算法）
- 自动认领高价值任务（>=40分）

### 2. 执行阶段（自动化）

**触发条件**: 队列中有 `claimed` 状态的任务

**执行顺序**:
1. 按评分降序排列（高分优先）
2. 按认领时间升序（先到先得）
3. 特殊优先：有明确金额的任务

**每个任务的完整流程**:
```
claimed → fork → clone → analyze → develop → test → commit → push → submit PR
```

### 3. 质量控制

**开发前**:
- ✅ 验证任务有效性
- ✅ 检查是否已有人提交PR
- ✅ 评估技术难度和可行性

**开发中**:
- ✅ 遵循项目代码规范
- ✅ 添加必要测试
- ✅ 更新相关文档

**提交前**:
- ✅ 本地测试通过
- ✅ 代码格式化
- ✅ commit message清晰

---

## 🔄 自动化触发点

### 定时触发
- **扫描**: 每30分钟（crontab）
- **开发**: 每次会话启动时检查队列
- **推送**: 每2小时（网络恢复后）

### 事件触发
- 新任务认领 → 立即开始开发
- PR被合并 → 开始下一个任务
- PR被拒绝 → 修复或跳过

---

## 📊 任务优先级算法

```python
优先级 = (
    评分 * 0.4 +
    金额权重 * 0.3 +
    时间权重 * 0.2 +
    技术匹配度 * 0.1
)

金额权重:
  $1000+ → 100分
  $500-999 → 80分
  $100-499 → 60分
  $50-99 → 40分
  <$50 → 20分
```

---

## 🚫 例外情况（需暂停询问）

1. **技术障碍**: 遇到无法解决的技术问题
2. **需要决策**: 多个实现方案需选择
3. **资源不足**: 需要API密钥、权限等
4. **风险评估**: 可能影响系统安全的操作
5. **用户干预**: 用户明确要求暂停

---

## 📈 效率目标

- **开发速度**: 每天完成3-5个任务
- **成功率**: 80%以上（PR被合并）
- **响应时间**: 认领后2小时内开始开发
- **完成时间**: 平均每个任务4-8小时

## 🔧 实施细节

### Git工作流
```bash
# 每个任务一个分支
git checkout -b fix/issue-{number}-{short-desc}

# 完成后合并
git checkout main
git merge fix/issue-{number}-{short-desc}
git push origin main
```

## ⚠️ 注意事项

1. **网络问题**: 本地先完成，网络恢复后批量推送
2. **并行限制**: 最多同时开发3个任务
3. **失败处理**: 连续失败3次 → 暂停并询问
4. **时间限制**: 单个任务超过8小时 → 评估是否继续
